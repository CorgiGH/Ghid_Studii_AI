What happens when you run a program?

When you double-click an executable or type a command in the terminal, the operating system doesn't just "run the code." It creates something called a process — a living, managed instance of that program.

A process has its own memory space, its own set of open files, and its own execution state. But here's the key insight: a process isn't always running on the CPU.

📖 Definition

Process: An instance of a program in execution. It includes the program code (text section), current activity (program counter, registers), stack, data section, and heap.

💡 Think about it

If you open the same program twice (e.g., two browser windows), how many processes are created? Are they the same process or different?

Click to reveal answer ▼
Two separate processes. Each has its own memory space, process ID (PID), and execution state. They share the same program code on disk, but in memory they are completely independent. If one crashes, the other keeps running.
Learn

Process States

A process moves through several states during its lifetime. At any moment, a process is in exactly one of these states:

  • New — Being created
  • Ready — Waiting to be assigned to a CPU
  • Running — Instructions are being executed
  • Blocked — Waiting for some event (I/O, signal)
  • Terminated — Finished execution
📈 Interactive Diagram — click a state to learn more
New: Process is being created. Memory is being allocated, PCB is initialized. Not yet in the ready queue.
Ready: Process has everything it needs to run and is waiting in the ready queue. The scheduler will pick it when a CPU is available.
Running: Process instructions are actively being executed on a CPU core. Only ONE process per core can be in this state.
Blocked (Waiting): Process is waiting for an external event — disk I/O, network response, user input. It cannot go directly back to Running, it must go through Ready first.
Terminated: Process has finished or was killed. PCB still exists until parent reads exit status (zombie state in UNIX).
💡 Think about it

A process is reading a large file from disk. What state is it in? When the disk finishes, does the process go straight to Running?

Click to reveal answer ▼
While waiting for disk I/O, the process is in the Blocked state. When I/O completes, it moves to Ready, NOT directly to Running. It must wait in the ready queue until the scheduler gives it CPU time. This is why the red dashed arrow in the diagram says "NOT allowed."
Learn

Process Control Block (PCB)

The OS needs to keep track of every process. It does this with a data structure called the Process Control Block (PCB) — think of it as the process's "ID card" that the OS carries around.

Every time a context switch happens, the OS saves the current process's state INTO its PCB, then loads the next process's state FROM its PCB.

📖 Definition

PCB contains:

  • PID — Unique process identifier
  • State — Current state (new, ready, running...)
  • Program Counter — Address of next instruction
  • CPU Registers — Saved register values
  • Memory Info — Page tables, segment tables
  • I/O Status — Open files, devices
  • Scheduling Info — Priority, queue pointers
💻 See it in Linux
$ cat /proc/self/status
Name:   cat
State:  R (running)
Pid:    12345
PPid:   12300
Threads: 1
VmSize: 8432 kB
VmRSS:  1204 kB
...

In Linux, /proc/[pid]/ exposes PCB-like information as files. Try it!

Context Switching

When the OS switches the CPU from one process to another, it performs a context switch. This is pure overhead — no useful work gets done during the switch.

The switch involves: saving the state of the current process → loading the state of the next process → jumping to the new process's program counter.

💡 Think about it

Context switches take ~1-10 microseconds. If a system does 1000 context switches per second, how much CPU time is "wasted" on overhead?

Click to reveal answer ▼
At 10µs per switch × 1000 switches = 10ms per second, which is only 1% overhead. Acceptable! But if switches take longer (e.g., heavy TLB flushes) or happen more frequently, it adds up. This is why efficient scheduling matters.
✅ Quick Check

Question 1: A process is waiting for keyboard input. What state is it in?

A
Ready — it's ready to run once input arrives
B
Blocked — it's waiting for an I/O event
C
Running — it's actively checking for input
✅ Blocked is correct! Keyboard input is I/O. The process cannot continue until the event occurs, so the OS puts it in the Blocked state and schedules another process.
✅ Quick Check

Question 2: What information is NOT stored in the PCB?

A
Program counter
B
Open file descriptors
C
The program's source code
✅ Source code is NOT in the PCB. The PCB stores runtime state (registers, PC, memory maps, I/O), not the original source. The compiled binary is in the text section of the process's memory.
