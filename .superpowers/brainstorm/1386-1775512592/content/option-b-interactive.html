What happens when you run a program?

Step 1 of 8 · Process Fundamentals

Learn

When you double-click an executable or type a command in the terminal, the operating system doesn't just "run the code." It creates something called a process — a living, managed instance of that program.

A process has its own memory space, its own set of open files, and its own execution state. But here's the key insight: a process isn't always running on the CPU.

🎓
Lecture 3, Slides 1–4
Prof introduces program vs. process distinction
📖 Definition

Process: An instance of a program in execution. It includes the program code (text section), current activity (program counter, registers), stack, data section, and heap.

💡 Think about it

If you open the same program twice (e.g., two browser windows), how many processes are created?

Click to reveal ▼
Two separate processes. Each has its own memory space, PID, and execution state. They share the same code on disk, but are completely independent in memory.

Process States

Step 2 of 8 · Process Fundamentals

Learn

A process moves through several states during its lifetime. At any moment, it's in exactly one state:

  • New — Being created
  • Ready — Waiting for CPU
  • Running — Executing on CPU
  • Blocked — Waiting for I/O
  • Terminated — Finished
⚠

Exam emphasis (Prof. Popescu): "Draw the complete state diagram with all transitions labeled. Students always confuse Blocked and Ready."

📈 Interactive — click a state
New: Memory is being allocated, PCB initialized. Not yet in the ready queue.
Ready: Has everything it needs. Waiting in queue for CPU time.
Running: Actively executing. Only 1 process per CPU core can be here.
Blocked: Waiting for I/O. Must go through Ready (not directly to Running) when done.
Terminated: Done. PCB lingers until parent reads exit status.

💡 Think about it

A process is reading from disk. What state is it in? When done, does it go straight to Running?

Click to reveal ▼
It's Blocked. When I/O completes, it goes to Ready, not Running. It must wait for the scheduler. This is the most common exam mistake.

Process Control Block (PCB)

Step 3 of 8 · Process Internals

Learn

The OS tracks every process using a Process Control Block (PCB) — the process's "identity card." It stores everything the OS needs to manage, pause, and resume a process.

💡 Think about it

Why does the PCB store the program counter? What would happen if it didn't?

Click to reveal ▼
Without saving the PC, a context-switched process would lose its place. When resumed, it wouldn't know which instruction to execute next — it would either restart from the beginning or crash. The PC is the "bookmark" for execution.
